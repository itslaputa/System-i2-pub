# System-I2: публичное руководство для агентов

[English](agents.md) | [Русский](agents.ru.md)

System-I2 — локальное desktop-приложение для учёта задач, проектов, категорий, комментариев и аналитики времени. Интерфейс построен на SolidJS/TypeScript/Vite, бэкенд — на Rust и Tauri 2, хранилище данных — SQLite.

Этот файл предназначен для публичных форков и внешних кодовых агентов. Не опирайтесь на приватные заметки мейнтейнера, локальные данные приложения или машинно-зависимые пути.

## Инварианты данных

- Приложение не поставляется с пользовательскими данными.
- Путь к данным берётся только из пользовательской настройки `runtime-bundle.json`.
- Для папок конфигурации, данных приложения и рабочего стола используйте системные API путей. Никогда не собирайте локальные пользовательские пути вручную.
- Отладочная конфигурация данных использует `System-I2-Dev`.
- Релизная конфигурация данных использует `System-I2`.
- Корректная папка данных содержит:
  - `tasks.sqlite3`
  - `task_categories.json`
  - `task_category_change_log.log`
  - `project_categories.json`
- Пользователь может подключить существующую папку данных, создать папку в выбранном месте или создать её в стандартной системной папке приложения.
- Игнорируемую папку `Data/` можно использовать для локальных данных во время разработки. Её содержимое нельзя коммитить.

## Правила модели данных

- ID категории задачи — стабильный идентификатор для аналитики.
- Название категории — изменяемый отображаемый текст.
- Проекты глобальны и не зависят от категорий задач.
- Один проект может содержать задачи из разных категорий.
- Категории проектов — необязательные метаданные типа проекта из `project_categories.json`; это не категории аналитики задач.
- Завершённые проекты скрыты из выбора проекта в форме задачи, но остаются видимыми и доступными для переоткрытия в каталоге и настройках.
- Суммы проектов, даты начала/окончания и состояние завершения выводятся из задач, триггеров и сервисной синхронизации. Не считайте эти значения ручным источником истины.

## Карта архитектуры

Интерфейс:

- `src/app/App.tsx`: проверка готовности данных и переключение между первым запуском и основным приложением.
- `src/app/AppShell.tsx`, `src/app/routes.ts`, `src/layout/`: оболочка приложения и навигация.
- `src/features/setup/`: первый запуск и настройка папки данных.
- `src/features/task-manager/`: запись задач, выбор проекта, каталог проектов.
- `src/features/analytics/`: блоки дашборда, графики, тренды и форматирование.
- `src/features/settings/`: редактор дерева категорий, редактор типов проектов, просмотр хранилища, журнал изменений, кнопка резервной копии.
- `src/services/tauri/`: тонкий TypeScript-мост к командам Rust.

Бэкенд:

- `src-tauri/src/lib.rs`: только регистрация Tauri-команд.
- `src-tauri/src/storage/`: подключение SQLite, схема, миграции, валидация и помощники для путей данных.
- `src-tauri/src/runtime/`: первый запуск, проверка папки данных, подключение/создание папки данных, zip-бэкап.
- `src-tauri/src/categories/`: JSON-дерево категорий задач и журнал изменений.
- `src-tauri/src/project_categories/`: плоский список типов проектов.
- `src-tauri/src/projects/`: CRUD проектов, статус и логика типов.
- `src-tauri/src/tasks/`: правила создания, чтения и удаления задач.
- `src-tauri/src/analytics/`: SQL-агрегации, дерево категорий, сводки проектов и тренды.

## Контракты Tauri-моста

Держите передаваемые TypeScript-данные согласованными с Rust serde-именами и сигнатурами команд.

Для изменений на границе фронтенда и бэкенда:

- обновляйте тесты Rust-команд и типов;
- обновляйте тесты моста и вспомогательных функций интерфейса;
- осознанно сохраняйте camelCase и serde naming;
- держите `src/services/tauri/` тонким мостом; доменную логику размещайте в модулях функций или бэкенд-модулях.

## Проверки

Фронтенд:

```bash
npm run test:frontend
npx tsc --noEmit --pretty false
npm run build
```

Бэкенд:

```bash
cargo test --manifest-path src-tauri/Cargo.toml
cargo check --manifest-path src-tauri/Cargo.toml
```

macOS `.app`:

```bash
npm run tauri build -- --bundles app
```

## Чеклист публичной безопасности

Перед публикацией или принятием PR проверьте:

- приватные пользовательские пути;
- файлы пользовательской конфигурации вроде `runtime-bundle.json`;
- SQLite-файлы, служебные sidecar-файлы, JSON-файлы данных и журналы данных;
- скопированные папки данных приложения;
- `node_modules/`, `dist/`, `src-tauri/target/` и сгенерированные схемы;
- приватные заметки мейнтейнера или локальные черновики.

Полезные проверки:

```bash
rg -n "private-path-marker|local-user-path-marker" README.md AGENTS.md agents_pub docs src src-tauri tests
find . \( -name "*.sqlite3" -o -name "*.sqlite3-*" -o -name "runtime-bundle.json" \) -print
```

## Частые ошибки

- Не заменяйте системные API путей на жёстко заданные локальные пути.
- Не используйте названия категорий как стабильный идентификатор аналитики; используйте ID категорий.
- Не связывайте проекты с категориями задач.
- Не показывайте завершённые проекты в выборе проекта формы задачи.
- Не скрывайте завершённые проекты из каталога и настроек, где их можно восстановить.
- Не коммитьте пользовательские данные только потому, что они удобны для локального тестирования.
- Не описывайте Windows/Linux-релизы как проверенные, если они не проходили дымовую проверку.

## Правила вклада

Держите изменения сфокусированными и покрывайте их тестами на правильной границе. Изменения документации не требуют тестов кода, если они не меняют команды, поведение сборки или опубликованные контракты.
